# Sessie 30 jun 2026 — VibeKilled.rip

## Wat is deze sessie gedaan

### Durable anonieme identiteit (cookie-anker) — "anoniem profiel bouwt nu op"

**Probleem (Thomas):** elke keer terugkomen + een paar acties = registreert als een andere naam.
Het anonieme profiel bouwde nooit op.

**Root cause:** identiteit zat *alleen* in localStorage (`vk:userId` + `vk:name`).
localStorage wordt gewist door Safari/iOS ITP (~7d), private mode, of cross-origin/andere
browser. Bij elke wipe mintte `getUserId()` een vers id en `getName()` een verse random
naam → nieuw "persoon" → server-side profiel (kills, medals, score, streak — alles aan
`userId` in Redis) accumuleerde nooit.

**Fix — server-set, durable first-party cookie + localStorage-mirror:**

- **`src/proxy.ts`** (nieuw — Next 16 hernoemde `middleware.ts` → `proxy.ts`, `export function proxy`):
  zet bij elk request `vk_uid` als server-`Set-Cookie` (Max-Age ~400d, sliding, `SameSite=Lax`,
  `Secure` alleen op https, `httpOnly:false` zodat client 'm leest). Bestaat de cookie al →
  alleen expiry verversen; nieuw device → één keer een vers `u_…` id (`crypto.randomUUID()`).
  Server-`Set-Cookie` overleeft ITP wél (anders dan localStorage én client `document.cookie`).
  Matcher: alle routes behalve `_next/static|_next/image|favicon.ico`.
- **`src/lib/identity.ts`**:
  - `getUserId()` = **localStorage-first → cookie → mint**, mirrort altijd naar **beide**.
    localStorage-first houdt bestaande profielen intact; cookie is de durable fallback.
  - `getName()` leidt default-naam **deterministisch** af uit het id (`devName(seedFromId(uid))`)
    → wiped naam-opslag geeft exact dezelfde alias terug. Custom naam blijft override.
  - `setName()` schrijft nu ook naar de cookie. Nieuwe helpers: `readCookie`/`writeCookie`/
    `seedFromId`, `COOKIES = { uid: "vk_uid", name: "vk_name" }` (cookie-naam mag geen ":").

**Server-side: niks gewijzigd** — profiel hing al volledig aan `userId` in Redis, dus het
persisteert nu automatisch zodra het id durable is.

## Beslissingen / notes
- Aanpak gekozen: **server-cookie + proxy** (durabel op Safari/iOS) i.p.v. alleen client dual-write
  (een client-set cookie wordt óók door ITP na ~7d gecapt).
- **Geen** canonical redirect: Thomas gebruikt in praktijk alleen `vibekilled.rip` (één origin).
- Bestaande users op Thomas' device verliezen niks: localStorage-first houdt het oude id vast en
  corrigeert de cookie ernaar op dezelfde load.
- AGENTS.md gevolgd: eerst Next 16 proxy-docs gelezen (`node_modules/next/dist/docs/...proxy.md`)
  vóór code — `middleware` → `proxy` was precies de breaking change.
